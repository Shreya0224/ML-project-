About This Project


  
  

    
🎯 Purpose & Problem Statement

    
Traditional linters (ESLint, PyFlake8, Checkstyle) detect syntax errors and style violations but provide no signal about the skill level of the developer or the conceptual gaps in their code. A beginner and an expert can both write syntactically valid Python — yet their code is radically different in structure, robustness, and idioms.

    
The Coding Skill Gap Detector bridges that gap by applying machine learning and program analysis to answer: "What level is this developer at, and what should they learn next?"

    
This tool is designed as a viva/demo-ready academic project demonstrating the full ML pipeline: data preparation → feature engineering → model training → inference → API → interactive UI.

  


  
  

    
👥 Who Uses It & Why

    

      
🎓
Students & Learners
Get instant, objective feedback on code quality without waiting for a tutor. Understand why their code is beginner-level and what specific habits to change next.

      
👩‍🏫
Educators & Bootcamps
Automatically grade or screen student submissions for structural quality — not just correctness. Identify which students need OOP, error handling, or testing skills.

      
🏢
Hiring Platforms
Pre-screen technical test submissions for skill level alignment before human review. Filter candidates who submit copy-pasted, unstructured, or beginner-level solutions.

      
🔍
Code Review Tools
Embed as a pre-commit hook or CI pipeline step to flag structural regressions — not just style — when juniors submit PRs without error handling or tests.

      
📊
Researchers
Study how code patterns correlate with developer experience. The TF-IDF embedding space can be extended with real corpora (GitHub, CodeChef) for publication-quality research.

      
🧪
ML Practitioners
A practical demonstration that NLP techniques apply directly to source code — TF-IDF on code characters outperforms word-level tokenisation for cross-language classification.

    

  


  
  

    
📅 When Is It Used?

    
The tool is most valuable at three moments in the learning lifecycle:

    
1. Before learning — Baseline assessment. A learner pastes their first attempt at a problem to understand their starting point and what gaps exist.

    
2. During learning — Iterative feedback. Each improved attempt is re-analysed to track progress on specific metrics (error handling, type hints, OOP).

    
3. Before submission / interviews — Final check. Confirm code meets intermediate/advanced structural standards before a technical interview or homework submission.

  


  
  

    
🧠 Algorithm — Step by Step

    

      
1

      

        
Input Preprocessing

        
Raw source code is received as a UTF-8 string via HTTP POST. It is stripped of leading/trailing whitespace. Language is normalised to lowercase. No tokenisation or AST parsing is performed — this is intentional (language-agnostic approach).

      

    

    

      
2

      

        
TF-IDF Vectorisation (Character N-grams)

        
Why character n-grams? Source code has no natural word boundaries across languages. def, func, fn, and function all mean "define a function" — but a word-level tokeniser would treat them as unrelated. Character n-grams (size 2–4) capture shared substrings across these keywords, indentation patterns, operator density, and punctuation style.


        TfidfVectorizer(analyzer="char_wb", ngram_range=(2,4), max_features=5000, sublinear_tf=True)


        sublinear_tf=True applies log(1 + tf) instead of raw term frequency, preventing tokens that appear many times (like spaces or braces) from dominating the vector.

      

    

    

      
3

      

        
Logistic Regression Classification

        
The TF-IDF sparse matrix is fed into a Logistic Regression classifier trained on 60+ labelled snippets across 8 languages. It outputs class probabilities for Beginner (0), Intermediate (1), and Advanced (2).


        Why Logistic Regression over alternatives?

        • vs SVM: LR provides probability outputs natively (Platt scaling required for SVM).

        • vs Random Forest: RF needs far more data to outperform LR on sparse TF-IDF space.

        • vs Neural Network: Overkill — the feature space is already rich from TF-IDF; a neural net would overfit on a 60-sample corpus.

        • L2 regularisation (C=1.0) prevents overfitting; lbfgs solver handles multi-class natively.

      

    

    

      
4

      

        
Rule-Based Feature Engine (20+ detectors)

        
The ML label is adjusted up or down based on deterministic regex/heuristic signals: functions, classes, error handling, imports, type hints, decorators, async patterns, generics, interfaces, unit tests, recursion. Each language has its own pattern set (e.g., Go's if err != nil, Ruby's begin/rescue, C++'s template<>).


        Why hybrid? Pure ML can mislabel a 3-line Python class as "Advanced" if the character patterns match training data. Rule-based signals provide hard guarantees — if there are no functions and no error handling, the score is bounded.

      

    

    

      
5

      

        
Cosine Similarity in TF-IDF Space

        
The user's code vector and a curated ideal reference vector (one per language) are compared using cosine_similarity(). This measures structural alignment — how similar the code's character patterns are to well-written idiomatic code.


        A score of 0.0 means completely dissimilar; 1.0 means identical character distribution. Typical beginner code scores 0.05–0.20; advanced code scores 0.40–0.75 against the reference.

      

    

    

      
6

      

        
Halstead Metrics (Approximated)

        
Full Halstead requires a complete language lexer. We approximate by counting unique operators (η₁), total operators (N₁), unique operands (η₂), and total operands (N₂) via regex.


        Derived: Volume = N·log₂(η) — information content. Difficulty = (η₁/2)·(N₂/η₂) — error-proneness. Effort = D·V — cognitive effort to write/understand.

      

    

    

      
7

      

        
Cyclomatic Complexity (McCabe)

        
Counts decision points in the code: if, elif, else, for, while, case, catch, except, rescue, &&, ||. Formula: CC = decision_points + 1. Values ≤5 indicate simple code; 6–10 is moderate; 11+ indicates code that is hard to test and maintain.

      

    

    

      
8

      

        
Maintainability Index

        
A composite metric from Microsoft Research:

        MI = 171 − 5.2·ln(Volume) − 0.23·CC − 16.2·ln(Lines)

        Normalised 0–100. Values above 85 indicate highly maintainable code. Below 65 signals refactoring is needed.

      

    

    

      
9

      

        
Score Computation (Hybrid 0–100)

        
35% from ML class probability (confidence in predicted label).

        65% from 11 rule-based binary signals: functions, error handling, classes, comments, imports, type hints, async/recursion/decorators, unit tests, comprehensions, interfaces, generics.


        This weighting ensures ML and rule-based signals are equally influential — neither can dominate alone.

🤔 Why These Specific Technology Choices?

    

      
🐍
Flask (not Django/FastAPI)
Flask is minimal — no ORM, no admin panel, no bloat. For a single-endpoint ML API, Flask lets the ML code be the hero, not the framework. FastAPI would add async complexity without benefit for a synchronous sklearn model.

      
📊
scikit-learn (not PyTorch/HuggingFace)
The task does not require deep learning. TF-IDF + Logistic Regression achieves strong separability on the feature space with zero GPU requirement, millisecond inference, and full interpretability. Adding CodeBERT would require 400MB+ model weights for marginal gain on a small corpus.

      
🔤
char_wb N-grams (not word tokens)
Source code is not natural language. Keywords differ across languages (def vs func vs function). Character n-grams are language-agnostic and capture indentation depth, operator density, and structural patterns without a per-language lexer.

      
📐
Halstead + Cyclomatic (not AST metrics)
Full AST parsing requires a language-specific parser per language (8 parsers). Regex-based approximations give 90% of the signal at 1% of the complexity. They are sufficient for educational feedback even if not production-grade static analysis.

      
🎨
Single HTML File (no React/Vue)
Zero build step. No npm install, no webpack, no node_modules. The entire 1,200-line frontend runs directly from a file:// URL or any static host. Portability beats modularity for a demo/viva project.

      
🔗
Cosine Similarity over Edit Distance
Edit distance (Levenshtein) is O(n²) and sensitive to formatting. Cosine similarity in TF-IDF space is O(n) and measures conceptual similarity — two differently formatted but structurally similar programs will score high, as expected.

    

  


  
  

    
⚠ Known Limitations & Future Work

    
Corpus size: 60+ synthetic samples is sufficient to demonstrate the pipeline but would be replaced by a real dataset (e.g., CodeNet, GitHub GH Archive) in production. More data → better ML generalisation.

    
No AST parsing: Regex-based detectors can be fooled by code in comments or strings. A production system would use language-specific parsers (tree-sitter, javalang, go/ast).

    
Improved code is rule-based: The "improved code" output applies fixed transformations. A true code improvement engine would require a large language model (GPT-4, Claude) or program synthesis techniques.

    
Future extensions: (1) Train on real GitHub repositories by skill tier. (2) Integrate CodeBERT embeddings as additional features. (3) Add plagiarism detection via pairwise cosine similarity. (4) Build a VS Code extension. (5) Add support for Swift, Kotlin, Rust, PHP.
